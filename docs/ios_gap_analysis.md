# iOS 客户端功能缺失分析报告

> 基于低保真原型 V1 与现有 iOS 实现（46 个 Swift 文件）的逐屏对比，交叉验证后端 29 个 API 端点能力边界

**严重程度**：P0=主闭环阻断 / P1=功能模块缺失 / P2=细节逻辑不完整 / P3=增强项

---

## Screen 01：进入层（Onboarding）

| # | 缺失项 | 等级 | 说明 |
|---|--------|------|------|
| 1.1 | **登录/注册系统** | P0 | `MockAuthProvider` 硬编码固定 UUID，无真实认证。原型要求手机号登录 + 邮箱/第三方。`AuthProvider` 协议已抽象但无实现。 |
| 1.2 | **初始阶段问答** | P1 | 原型 Step 3 要求"当前关注"标签选择 + "近况一句话"自由文本，用于生成初始阶段判断。iOS 仅有 FocusTheme 多选，缺自由文本输入和基于回答的智能阶段判断。 |
| 1.3 | **可延后项（高压场景/作息/通知授权）** | P2 | 原型 Step 4 全部未实现。iOS 无延后补充入口或提醒机制。 |
| 1.4 | **通知权限请求** | P2 | 未调用 `UNUserNotificationCenter.requestAuthorization`，未注册推送令牌到后端 Device 表。后端有 Device 模型但**也缺少注册 API**。 |

## Screen 02：首页（Home / Status）

| # | 缺失项 | 等级 | 说明 |
|---|--------|------|------|
| 2.1 | **"当前周焦点"主卡信息不完整** | P1 | 原型要求展示阶段标签、本周主题、风险状态、一句话建议。iOS 仅展示儿童昵称+月龄，未渲染 `activePlan.focusTheme`、`primaryGoal`、`child.riskLevel`。`HomeSummaryResponse` 已返回这些字段。 |
| 2.2 | **"待处理回流"摘要卡** | P1 | 原型要求首页展示"周反馈已生成"等待处理项摘要。iOS 仅有消息 Tab 角标，首页无回流摘要。后端已返回 `weeklyFeedbackStatus/weeklyFeedbackId`。 |
| 2.3 | **今日任务拆分展示** | P2 | 原型拆分"主练习"和"自然嵌入"两栏展示。iOS 有跳转但缺分栏摘要。 |
| 2.4 | **记录与计划关联说明** | P2 | 原型要求标注"影响本周计划理解"和"已同步到计划页"。iOS 未展示关联。 |
| 2.5 | **儿童月龄自动刷新** | P2 | 未在启动时调用 `POST /children/{id}/refresh-stage`，跨月后月龄可能过时。 |

## Screen 03：计划页（Plan / Action）

| # | 缺失项 | 等级 | 说明 |
|---|--------|------|------|
| 3.1 | **"完成后去记录"桥梁** | P0 | 原型定义的**关键桥梁**：计划→记录连通。iOS 的 DayTaskDetailView 完成状态更新后无任何导航。`RecordCreate.source_plan_id` 从未传递。 |
| 3.2 | **"现在求助"上下文入口** | P1 | 原型要求计划完成面板旁放"现在求助"。iOS 计划页无此入口。浮动按钮不携带 `plan_id`，后端 `InstantHelpRequest` 支持此参数。 |
| 3.3 | **"周反馈与历史"入口** | P1 | 原型要求计划页底部提供"查看本周反馈"和"查看历次计划"。iOS 计划页无这两个导航。 |
| 3.4 | **`observation_candidates` 快速打点** | P2 | AI 生成的 5-8 个打点候选项未渲染。PlanResponse 含此字段但 UI 未展示。 |
| 3.5 | **`weekend_review_prompt` 周末复盘** | P2 | 后端 Day 6-7 复盘引导文本未展示。 |
| 3.6 | **`conservative_note` 保守路径** | P2 | 后端"更温和替代路径"说明未渲染。 |

## Screen 04：记录页（Record / Evidence）

| # | 缺失项 | 等级 | 说明 |
|---|--------|------|------|
| 4.1 | **语音记录入口** | P1 | `RecordType.voice` 枚举存在但无 UI。需集成 AVAudioRecorder → 上传 → 传 voice_url 完整链路。后端支持 `voice_url` + `transcript`。 |
| 4.2 | **快速打点（预设标签一键提交）** | P1 | 原型定义为记录页最快入口：预设标签点选后一键提交。iOS 的 RecordCreateView 是完整表单，缺少"一键提交"快捷路径。应联动 Plan 的 `observation_candidates` 作为候选标签。 |
| 4.3 | **记录来源关联** | P2 | `RecordCreate` 支持 `source_plan_id` 和 `source_session_id` 用于证据链关联，iOS 从未传递。 |
| 4.4 | **时间线中的关联说明** | P2 | 原型要求时间线条目标注"已同步到计划页，影响本周判断"。iOS 列表无此关联信息。 |

## Screen 05：即时求助（AI in Context）

| # | 缺失项 | 等级 | 说明 |
|---|--------|------|------|
| 5.1 | **AI 结果结构化渲染** | P0 | 后端返回的 `result` JSON 包含 `first_say`（先说什么）、`then_do`（接着做什么）、`fallback`（没接住怎么办）三段式结构。iOS 的 InstantHelpView 对 `AnyCodable result` 仅显示硬编码占位文本"AI 已生成个性化指导建议..."，**完全未解析渲染实际内容**。 |
| 5.2 | **上下文透明化展示** | P1 | 原型要求显式展示"系统已引用的上下文"（年龄、当前计划、最近记录）。后端返回 `context_snapshot` JSON，iOS 未解析展示。 |
| 5.3 | **后续回流动作** | P1 | 原型要求三个回流按钮：补记为记录 / 加入本周关注 / 查看咨询准备建议。iOS 展示 AI 结果后无任何后续动作入口，求助结果"悬空"。`RecordCreate.source_session_id` 可关联来源。 |
| 5.4 | **降级结果的语义化展示** | P2 | iOS 有降级结果文本展示但未做差异化样式处理。原型未明确但从安全角度应有视觉区分。 |

---

## Screen 06：消息回流（Message / Return）

| # | 缺失项 | 等级 | 说明 |
|---|--------|------|------|
| 6.1 | **消息深链回流** | P1 | 原型核心定义：消息的主按钮直接跳到目标页（计划/记录/周反馈），不只是打开消息正文。后端 `MessageResponse` 包含 `targetPage` 和 `targetParams` 字段用于精确导航，iOS 的 MessageListView 点击消息后仅标记已读+上报 clicked，**未实现根据 targetPage 进行页面跳转**。 |
| 6.2 | **消息类型图标/语义区分** | P2 | iOS 已有基于 `MessageType` 的图标和颜色区分，基本对齐。但缺少"处理状态"统计面板（未读/已深链/待补记 KPI）。 |
| 6.3 | **推送集成** | P1 | 完全未集成 APNs。后端有 PushProvider 接口+MockPushProvider，但缺少 Device 注册 API。iOS 端需要：注册推送→获取 token → 注册到后端 → 接收并处理推送 → 深链到目标页。 |

## Screen 07：周反馈（Weekly Feedback）

| # | 缺失项 | 等级 | 说明 |
|---|--------|------|------|
| 7.1 | **"本周积极变化"和"仍需多给机会"结构化展示** | P1 | 后端返回 `positive_changes`（JSON 数组）和 `opportunities`（JSON 数组），iOS 的 FeedbackView 仅展示 `summary_text` 文本段，未解析这两个结构化字段为分区卡片。 |
| 7.2 | **"更保守的下一步"边界说明** | P1 | 原型明确要求单独显式标记"如果多场景持续升级，转去咨询准备页"。后端返回 `conservative_path_note`，iOS 未渲染。 |
| 7.3 | **决策选项与下周计划联动** | P2 | iOS 已有决策提交 UI，但提交成功后未触发下周计划自动生成或导航到新计划页。原型要求"继续这一周主线"按钮应无缝过渡到下周行动。 |
| 7.4 | **记录统计与完成率展示不完整** | P2 | 后端返回 `record_count_this_week` 和 `completion_rate_this_week`，iOS FeedbackView 有展示但样式为简单文本，缺少原型要求的统计概览卡片格式。 |

## 跨页面系统性缺失

| # | 缺失项 | 等级 | 说明 |
|---|--------|------|------|
| S.1 | **计划→记录→反馈主闭环断裂** | P0 | 原型定义的核心闭环"计划执行→记录留证→反馈回流"在 iOS 中三段均为独立页面，缺乏导航桥梁（3.1）和来源关联（4.3）。用户完成计划任务后无引导去记录，记录后无引导看反馈。 |
| S.2 | **AI 结果无法回流到系统** | P0 | 即时求助结果（5.3）和周反馈决策（7.3）均无法将 AI 输出转化为系统内的下一步动作（记录/计划/关注点更新）。 |
| S.3 | **`AnyCodable` JSON 字段普遍未解析** | P1 | `result`、`context_snapshot`、`positive_changes`、`opportunities`、`decision_options`、`observation_candidates` 等 AnyCodable 字段在 Model 层已声明，但 View 层几乎全部未解析为具体 UI 组件。需要定义强类型 Swift 结构体来映射这些 JSON。 |
| S.4 | **风险升级的前端感知缺失** | P1 | 后端在 AI 即时求助中检测到 `suggest_consult_prep=true` 时会自动升级 child.risk_level 并创建 risk_alert 消息。iOS 未特殊处理 risk_alert 类型消息，也未在 child 信息变化后刷新首页状态。 |
| S.5 | **多儿童切换后数据不刷新** | P2 | AppState 支持 `activeChildId` 切换，但 HomeView/PlanDetailView/RecordListView 的 ViewModel 在 childId 变化时未重新加载数据。 |
| S.6 | **网络状态/离线提示缺失** | P3 | 无 NWPathMonitor 或类似的网络可达性监听。 |

## 后端侧配套缺口

| # | 缺失项 | 说明 |
|---|--------|------|
| B.1 | **Device 注册 API 缺失** | 后端有 Device ORM 模型但无 CRUD 端点，iOS 无法注册推送令牌。 |
| B.2 | **文件上传 API 缺失** | 语音记录需要上传音频文件，后端无文件上传接口（OSS presigned URL 或 multipart upload）。 |
| B.3 | **历次计划列表 API 缺失** | 原型"查看历次计划"需要按 childId 获取历史计划列表，后端仅有 `GET /plans/active` 和 `GET /plans/{id}`，无 list 接口。 |

## 优先级排序建议

**第一批（P0 — 主闭环连通）**：

1. **3.1** 计划完成后→记录桥梁导航 + source_plan_id 传递
2. **5.1** AI 结果结构化渲染（定义 Swift 强类型 → 解析 AnyCodable → 三步卡片 UI）
3. **S.1** 主闭环导航打通（计划→记录→反馈→下周计划的完整流转）
4. **S.2** AI 结果回流动作（补记为记录、加入关注）

**第二批（P1 — 模块完整性）**：

5. **2.1 + 2.2** 首页周焦点主卡 + 回流摘要卡
6. **4.1 + 4.2** 语音记录入口 + 快速打点
7. **5.2 + 5.3** AI 上下文展示 + 后续回流动作
8. **6.1 + 6.3** 消息深链回流 + 推送集成
9. **7.1 + 7.2** 周反馈结构化展示 + 保守路径
10. **3.2 + 3.3** 计划页求助入口 + 周反馈/历史入口
11. **S.3** AnyCodable 字段统一强类型化
12. **S.4** 风险升级前端感知

**第三批（P0 但工程量大，可并行推进）**：

13. **1.1** 登录/注册系统（依赖 JWT/OAuth 技术选型）

**第四批（P2 — 细节补全）**：

14. 其余 P2 项按所属页面分批实现

## 统计汇总

| 等级 | 数量 | 占比 |
|------|------|------|
| P0 — 阻断 | 6 项（含系统性） | 19% |
| P1 — 重要 | 16 项 | 50% |
| P2 — 补全 | 10 项 | 31% |
| **合计** | **32 项** | 100% |

> 当前 iOS 客户端已完成了**页面骨架和 API 通信层**的搭建（27 个端点 1:1 对齐、12 个枚举对齐、MVVM 架构完整），但在**主闭环连通性**（计划→记录→反馈）、**AI 结果渲染**和**消息深链回流**三个维度存在实质性缺失，这正是原型重点验证的三件事。

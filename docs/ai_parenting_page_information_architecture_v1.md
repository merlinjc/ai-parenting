# 18—48个月幼儿家长辅助型 AI 产品：页面信息架构 V1

## 一、文档目标

本稿承接《产品设计草案》《7 天微计划模板 V1》与《整体服务架构草案 V1》，目的不是进入视觉稿或交互细节，而是先把 **iOS 家长端的页面信息结构** 固定下来。换句话说，本稿要回答的核心问题并不是“页面长什么样”，而是 **家长在 App 中先看到什么、下一步去哪里、不同页面分别承接哪类服务状态，以及消息与 AI 如何回流到正确的任务入口**。

之所以现在进入页面信息架构，而不是直接画原型，是因为前面的产品定位、观察模型、微计划模板与服务架构已经足够稳定。当前更需要做的，是把这些后端能力和内容能力翻译成前台可理解、可执行、可回流的页面体系。只有先把页面层级、导航关系与状态映射理顺，后续视觉设计、字段设计和交互细化才不会反复返工。

| 本稿解决的问题 | 本稿暂不展开的问题 |
|---|---|
| 明确 iOS 端主导航与主页面集合 | 页面视觉风格与组件级交互 |
| 固定首页、计划、记录、消息、即时求助等页面职责 | 接口字段与数据库表结构 |
| 说明页面如何映射到服务模块与核心对象 | 动效、插画与品牌表达 |
| 明确推送、AI 输出与记录状态的回流路径 | 可用性测试与高保真原型 |

## 二、页面信息架构的设计前提

这款产品的页面结构必须继续服从既定产品边界：**服务对象是家长，而不是儿童；产品核心是帮助家长稳定观察、记录、执行与复盘，而不是鼓励家长频繁与 AI 长对话；前台结构应优先服务真实家庭互动，而不是制造内容消费时长。** 因此，页面信息架构不能采用资讯流优先、聊天优先或“功能越多越平铺”的常见做法。

对于当前产品，页面层最重要的任务有三个。第一，是让家长快速知道“这周最该关注什么、今天最先做什么”。第二，是让记录和计划之间形成清楚的前后关系，使家长感到自己的记录确实影响了系统后续建议。第三，是让 AI 与消息都变成 **上下文驱动的辅助能力**，而不是脱离业务的独立内容区。

| 架构前提 | 对页面层的直接要求 |
|---|---|
| **家长端主入口** | 页面语言、任务组织与状态提示都面向照护者理解和执行 |
| **记录先于建议** | 记录入口必须高可达，且能被首页与计划页持续引用 |
| **计划是主行动骨架** | 7 天微计划应成为一条清晰的独立信息主线 |
| **AI 是上下文能力** | AI 不应被设计成无边界聊天首页，而应嵌入具体场景 |
| **消息承担回流** | 通知与消息必须落到具体任务页，而不是停留在列表层 |
| **非诊断化表达** | 页面文案与结构都要避免“评估结论页”式体验 |

## 三、页面结构备选方案与推荐结论

在进入具体页面之前，先需要固定主导航策略。基于当前产品定位，至少有三种常见做法。第一种，是将 **AI 对话** 设为底部主 Tab，与首页、计划、记录并列；第二种，是以 **首页—计划—记录** 作为主骨架，再把消息、档案和即时求助设计为上下文入口；第三种，是采用 **首页单页聚合**，把大部分能力收纳进首页卡片，通过深链进入详情。

综合当前产品阶段、低屏幕原则与家长使用心智，推荐采用第二种：**“三大主任务页 + 两类辅助回流页 + 一组账户与设置页”**。其中，首页、计划、记录构成日常高频主骨架；消息中心与即时求助承担事件回流和高压场景支持；家庭档案与设置则作为次级支撑层存在。这样既能保证页面结构稳定，也能避免把 AI 错误地塑造成产品中心。

| 方案 | 结构特点 | 优点 | 风险 | 结论 |
|---|---|---|---|---|
| **方案 A：AI 主 Tab** | 首页 / 计划 / 记录 / AI / 我的 | AI 入口非常显眼，开发上直观 | 容易把产品重心误导为聊天工具 | 不推荐 |
| **方案 B：任务主骨架 + 上下文入口** | 首页 / 计划 / 记录 为主，消息与 AI 为回流入口 | 最符合“观察—计划—支持—回流”的产品链路 | 需要更严格地设计入口优先级 | **推荐** |
| **方案 C：首页大聚合** | 多数功能收进首页卡片 | 初期页面数量少 | 结构很快膨胀，长期难以维护 | 不推荐 |

## 四、推荐的整体页面分层

基于推荐方案，iOS 端信息架构建议固定为三层：**进入层、主任务层、支撑层。** 进入层负责用户进入系统并完成建立上下文的动作；主任务层承接日常最高频行为；支撑层承接消息回流、即时求助、家庭档案与设置等次级但必要能力。这样的层次能把“首次进入”“日常使用”“偶发回流”清楚区分开。

进入层不应过重，但必须完成后续个性化所需的最小前置条件，例如账户登录、儿童基本信息、照护者角色、阶段初筛与通知授权。主任务层则应尽量稳定，避免频繁改动底部骨架。支撑层可以根据场景由首页、推送或主页面中的入口触发，不必在第一版全部提升为底部主导航。

| 页面层级 | 页面集合 | 主要作用 |
|---|---|---|
| **进入层** | 启动页、登录注册、儿童档案建立、阶段起始问答、通知授权 | 完成身份建立与基础上下文输入 |
| **主任务层** | 首页、计划页、记录页 | 承接家长日常最核心的查看、执行与输入动作 |
| **支撑层** | 即时求助、消息中心、周反馈详情、计划历史、档案与设置、风险说明与咨询准备页 | 承接回流、复盘、次级管理与边界说明 |

## 五、主导航建议：三大主任务页 + 全局辅助入口

主导航建议采用 **底部三 Tab**：**首页、计划、记录**。这三个一级入口分别对应“当前状态总览”“本周行动组织”“证据输入与回看”三类最稳定的家长任务。这样做的关键，不是为了减少页面数量，而是为了让用户每次打开 App 时都能迅速判断：我是来看看现在情况、执行本周动作，还是补一条记录。

消息中心、即时求助和档案设置不建议放入同级主 Tab。消息中心虽然重要，但其本质是回流型入口，更适合通过顶部消息按钮、未读徽标和推送跳转进入。即时求助也不应被误解为产品主导航，而应通过首页快捷入口、计划页上下文按钮、记录提交后的延伸入口与全局悬浮辅助按钮进入。**AI 的职责是缩短家长从问题到动作的路径，而不是增加停留在聊天中的时长。** 档案与设置则属于低频管理页，适合在首页右上角或个人入口中承接。与此同时，需要继续钉住首页与计划页的边界：**首页负责判断与跳转，计划页才是完整执行页；首页不展开承载完整任务细节。**

| 导航元素 | 位置建议 | 职责 |
|---|---|---|
| **首页** | 底部主 Tab 1 | 展示当前阶段状态、今日任务、最近记录、待处理回流 |
| **计划** | 底部主 Tab 2 | 承接 7 天微计划、日任务、完成状态与周反馈 |
| **记录** | 底部主 Tab 3 | 承接轻量打点、事件记录、语音转写、周回顾与历史回看 |
| **消息中心** | 顶部按钮 + 推送深链 | 承接提醒、结果通知与状态回流 |
| **即时求助** | 全局快捷按钮 + 上下文 CTA | 承接“现在怎么办”类高压场景支持 |
| **档案与设置** | 首页右上角 / 次级入口 | 承接儿童档案、照护者信息、通知偏好与账户设置 |

## 六、进入层信息架构

进入层的目标不是完成冗长 onboarding，而是用最少步骤建立后续页面能够成立的上下文。建议入口链路保持为：**启动页 → 登录 / 注册 → 儿童档案建立 → 照护者角色与家庭场景补充 → 初始阶段问答 → 通知权限引导 → 进入首页。** 这样一来，首页第一次出现时，就已经有了月龄阶段、关注主题和最初的计划与记录框架，不会让用户面对空页面。

进入层尤其要避免两个问题。第一，不要在第一次启动时一次性收集过多资料，否则会显著抬高进入门槛。第二，不要在通知权限之前完全不给出价值说明，否则家长难以理解为什么允许消息触达。更合理的做法是，在儿童档案与阶段问答结束后，用“每日提醒 / 周反馈生成 / 计划回流”这样的明确价值说明，再引导开启通知。进入首页的最小必填项，应只包含 **账户建立、儿童基础信息与初始阶段判断**；家庭场景补充与通知授权则可以延后完成，以避免首次进入被不必要地阻断。

| 进入页 | 核心信息 | 进入下一步的条件 |
|---|---|---|
| **启动页** | 品牌与状态恢复 | 判断是否已有有效登录态 |
| **登录 / 注册** | 手机号、邮箱或第三方登录 | 成功建立账户与会话 |
| **儿童档案建立** | 出生年月、昵称、主要照护关系 | 形成阶段判断基础 |
| **家庭场景补充** | 高压场景、常见作息、主要照护者 | 支撑计划与 AI 上下文 |
| **初始阶段问答** | 当前关注主题、近况描述、重点困扰 | 生成初始焦点与首页卡片 |
| **通知授权引导** | 说明提醒与回流价值 | 建立推送能力 |

## 七、首页信息架构

首页应被定义为 **状态首页**，而不是内容首页。它的作用是让家长在最短时间内回答四个问题：**孩子当前处于哪个阶段、这周主要关注什么、今天最先做什么、最近是否有需要处理的反馈或提醒。** 因此，首页不适合采用资讯流式布局，也不应把大量解释性内容铺在第一屏。

建议首页以“当前周焦点”作为主卡，其下依次承接“今日任务”“最近记录反馈”“待处理消息 / 周反馈”和“即时求助快捷入口”。如果当前没有活跃计划，首页的第一优先级应变成“为什么现在建议开始一份计划”或“先补一条记录”；如果处于建议咨询状态，则首页的主卡应切换为更保守的支持与下一步说明，而不是继续鼓励强化训练。

| 首页模块 | 页面职责 | 对应服务能力 |
|---|---|---|
| **阶段与当前焦点卡** | 告知当前阶段、本周主题与状态级别 | 儿童档案 + 微计划模块 |
| **今日任务卡** | 呈现今天最优先的 1—2 个动作与完成状态 | 微计划与任务模块 |
| **最近记录摘要** | 告知最近是否已有新记录、是否影响计划理解 | 观察与记录模块 |
| **周反馈 / 待处理回流卡** | 呈现已生成反馈、未读提示或待补动作 | AI 编排 + 消息模块 |
| **即时求助快捷入口** | 在高压时刻快速进入上下文支持 | AI 会话与编排模块 |
| **消息与档案入口** | 进入消息中心或个人信息页 | 消息模块 + 账户档案模块 |

## 八、计划页信息架构

计划页应是产品最完整的 **行动页**。如果说首页负责告诉用户“现在最重要的是什么”，那么计划页就负责把“这一周具体怎么做”讲清楚。页面结构建议以 **当前 7 天计划** 为中心，向下展开“本周家长目标”“Day 1—Day 7 节奏”“当日任务详情”“观察记录点”“周末总结入口”和“历史计划回看”。

这里尤其需要保持两个判断。第一，计划页不是课程页，不应出现大量和执行脱节的长篇理论说明。第二，计划页也不是打卡工具页，不应只剩勾选完成而丢失场景、话术与复盘信号。更适合的结构，是把执行动作、示范话术、有效信号与完成状态放在同一个页面体系中，让家长感到“我不是只在完成任务，而是在看见一个焦点是否真的出现变化”。在信息架构上，**周反馈详情属于计划链路的延展页面**，但可以被首页与消息中心直接深链唤起，以保证反馈既属于行动闭环，又具备稳定回流能力。

| 计划页层级 | 主要内容 | 页面目标 |
|---|---|---|
| **计划首页** | 当前计划标题、主目标、优先场景、周进度 | 告知本周主线与整体节奏 |
| **当日任务详情** | 主练习、自然嵌入机会、示范话术、记录点 | 支撑当天真实执行 |
| **完成与复盘面板** | 完成状态、是否顺利、是否需调整 | 形成轻量回写 |
| **周反馈详情** | 本周积极变化、仍需机会的点、下周建议 | 承接 AI 生成结果 |
| **计划历史** | 历次焦点、完成情况、周末结论 | 支撑长期回看与趋势理解 |

## 九、记录页信息架构

记录页是整个产品的 **证据输入页**。其核心不是让家长填表，而是让家长以极低负担捕捉“刚刚发生了什么”。因此，记录页应优先围绕三种输入组织：**轻量打点、关键事件记录、语音 / 短文本补充**。这些输入完成后，系统再在后台做结构化归档，而不应把复杂分类负担全部前置给用户。

记录页建议采用“快速入口 + 历史回看”双区结构。上半区提供当下可立即提交的轻量入口，例如“今天是否完成计划动作”“出现了哪类重点表现”“记录一个卡住场景”；下半区则提供最近记录时间线、按主题查看和周回顾入口。这样既能满足高频输入，也能保证家长逐步看到记录的累积价值。这里还需要明确与即时求助页的分工：**当家长需要当下回应方案时，应优先进入即时求助；当家长只是补充已经发生的事实、希望留下证据时，应优先进入记录页。**

| 记录页模块 | 主要动作 | 与其他页面的关系 |
|---|---|---|
| **快速打点入口** | 勾选今日状态、完成度或是否出现重点信号 | 回写首页与计划页 |
| **关键事件记录** | 记录一句话事件、发生场景、当时反应 | 进入趋势聚合与 AI 上下文 |
| **语音 / 文本补充** | 低负担记录复杂情境 | 后端转写并归档 |
| **周回顾入口** | 汇总一周内重要变化与困扰 | 为周反馈与下周计划提供输入 |
| **记录时间线** | 查看最近输入、按主题过滤 | 支撑复盘与连续性理解 |

## 十、即时求助页信息架构

即时求助页不应被设计为普通聊天页，而应被定义为 **场景化支持页**。用户进入这个页面，通常不是为了闲聊，而是因为在某个瞬时情境里需要“先说什么、接下来做什么、如果没接住怎么办”的支持。因此，页面结构应围绕 **当前问题场景、已有上下文、AI 给出的三步支持、必要的补充记录与后续动作** 来组织。

为了避免无边界对话，即时求助页建议优先提供场景卡入口，例如“转场崩溃”“不肯开口说”“外出见人就躲”“总是抢回合”这类常见问题；同时保留自由输入，但自由输入后也应被整理为结构化回答卡，而不是持续聊天气泡流。页面底部可提供“补记为记录”“加入本周关注”“仍未缓解，查看咨询准备建议”等后续动作，使求助结果回流到系统状态中。

| 即时求助页模块 | 页面职责 | 对应后端能力 |
|---|---|---|
| **问题场景选择 / 输入区** | 承接家长当下问题 | AI 会话入口 |
| **上下文提示卡** | 告知系统已引用哪些儿童阶段、近期记录或当前计划 | 上下文组装 |
| **三步支持结果卡** | 展示先说什么、接着做什么、无效时怎么调整 | AI 结果结构化 |
| **追加动作区** | 补充记录、关联到计划、查看风险提示 | 会话回写 + 记录 / 计划模块 |
| **边界说明区** | 提示非诊断化与咨询升级条件 | 风控与边界模块 |

## 十一、消息中心与回流页面信息架构

消息中心的价值不在“消息数量”，而在于 **把系统主动触达变成可执行的回流动作**。因此，消息中心不应只是一个静态列表，而应按回流目标组织，例如“去完成今日任务”“去查看周反馈”“去补一条记录”“去查看咨询准备建议”。这样，消息详情页也不应停留在消息文本本身，而要明确告诉用户点击后会发生什么。

页面层建议将消息分为四类：计划提醒、记录提醒、反馈生成、边界升级提示。消息详情页则根据类型分别落向计划详情、记录入口、周反馈详情或咨询准备页。未读标记、送达状态与点击回流可以继续由消息模块维护，但前台只需向用户展示最少必要的信息。与此同时，需要把消息中心与首页待处理卡的关系写清：**首页只展示当前最需要处理的回流事项，消息中心保留完整通知列表与处理状态。**

| 消息类型 | 推荐落地页 | 页面目标 |
|---|---|---|
| **今日任务提醒** | 计划页 / 当日任务详情 | 回到执行动作 |
| **周反馈已生成** | 周反馈详情页 | 促使家长阅读与进入下周决策 |
| **记录补充提醒** | 记录页 / 周回顾入口 | 补齐证据输入 |
| **风险或咨询提示** | 咨询准备页 / 档案说明页 | 引导进入更保守的下一步 |

## 十二、档案、设置与低频管理页面

虽然档案与设置不属于当前产品的高频核心任务，但它们是所有页面稳定运行的基础。建议将其组织为一个低频管理层，包含 **儿童档案、照护者信息、通知偏好、账号与隐私设置、帮助与边界说明**。这些页面不必进入主骨架，却必须保证在首页或全局位置稳定可达。

对于这款产品而言，档案页不仅是“个人资料”，更承担后续个性化上下文的维护职责。例如儿童年龄阶段变化、照护者切换、通知偏好调整，都会影响首页、计划与消息中心看到的内容。因此，虽然它不高频，但应保持结构稳定、信息清楚，并尽量减少多入口重复编辑。

| 低频管理页 | 主要内容 | 对系统的影响 |
|---|---|---|
| **儿童档案** | 出生年月、阶段信息、关注主题摘要 | 影响阶段判断与计划适配 |
| **照护者信息** | 与儿童关系、共同照护者角色 | 影响话术与推送触达逻辑 |
| **通知偏好** | 计划提醒、反馈提醒、静默时段 | 影响消息调度 |
| **账号与隐私** | 登录方式、数据说明、导出与注销 | 影响账户与合规边界 |
| **帮助与边界说明** | 非诊断化原则、何时建议咨询 | 影响风险表达一致性 |

## 十三、关键跨页跳转与状态回流

页面信息架构的关键，不只是列出页面集合，而是定义 **页面之间如何形成连续状态流转**。对本产品来说，主闭环可以被概括为：**家长在计划页执行动作，在记录页留下证据，系统基于记录生成反馈或提醒，再通过首页与消息把家长拉回下一次计划执行。** 在这个总闭环之下，至少还有五条必须明确的跨页链路：从首页进入计划执行、从计划完成后补记录、从记录触发 AI 支持、从消息回流到具体任务页，以及从风险提示进入咨询准备页。只要这五条路径清楚，App 的主闭环就基本成立。

更重要的是，每一次跳转都应伴随状态变化。例如计划页完成一个任务后，首页的今日任务卡应立即更新；记录提交后，计划页应知道“今天已有相关证据”；消息被点击后，消息中心要记录已处理；即时求助结果若被标记为“加入本周关注”，则下周计划生成时应能引用这条上下文。这种回写关系正是页面层与服务架构层对齐的地方。

| 关键链路 | 起点 | 终点 | 需要回写的状态 |
|---|---|---|---|
| **查看本周行动** | 首页 | 计划页 | 当前计划已查看 |
| **完成动作后补记** | 计划页 | 记录页 | 当日任务完成 + 新记录提交 |
| **高压场景求助** | 首页 / 计划 / 记录 | 即时求助页 | 新会话建立、结果引用上下文 |
| **消息回流执行** | 消息中心 / 推送 | 计划页 / 记录页 / 周反馈页 | 消息已读、已点击、回流目标已到达 |
| **风险升级处理** | 首页 / 消息 / 周反馈 | 咨询准备页 | 状态切换为更保守路径 |

## 十四、页面与服务模块的映射关系

为了让后续接口设计不脱节，页面信息架构需要继续明确“前台页面看到的到底是哪类后端对象”。在当前体系里，首页主要消费聚合态；计划页主要消费微计划对象；记录页主要操作观察记录对象；即时求助页主要操作 AI 会话对象；消息中心主要操作通知对象；档案与设置则主要消费用户、设备与儿童档案对象。

这种映射关系的重要性在于，它能防止页面层为了方便展示而任意拼接不稳定数据。只要页面职责与后端对象一一对应，后续即使进入字段设计，也能保持结构清楚、状态一致。

| 页面 | 主要服务模块 | 核心对象 | 主要读写行为 |
|---|---|---|---|
| **首页** | 微计划、记录、消息聚合 | 当前计划、最近记录、未读消息 | 读聚合状态、触发跳转 |
| **计划页** | 微计划与任务 | 微计划、日任务、周反馈 | 读计划、写完成状态、读反馈 |
| **记录页** | 观察与记录 | 观察记录、周回顾 | 写记录、读时间线、触发聚合 |
| **即时求助页** | AI 会话与编排 | 会话、结构化建议、风险提示 | 写提问、读结果、回写上下文 |
| **消息中心** | 通知与消息 | 消息通知、送达 / 点击状态 | 读列表、写已读、触发回流 |
| **档案与设置** | 账户、设备、儿童档案 | 用户、设备、儿童档案 | 读写账户与偏好信息 |

## 十五、V1 页面骨架的优先级建议

如果下一步进入原型或需求拆解，建议不要同时把所有页面做到同等精细。更合理的推进方式，是先把 **首页、计划页、记录页、即时求助页、消息回流逻辑** 作为第一优先级，因为它们共同构成了 MVP 的主闭环。档案设置、历史计划、帮助中心和咨询准备页可以在第二优先级继续细化。

这种优先级安排的原因并不复杂：当前产品最需要先验证的，是家长是否能顺畅地完成“看到重点—执行动作—留下一条记录—在高压时得到支持—再被消息拉回”这一整条路径，而不是所有低频管理功能是否足够完备。只要主闭环先顺了，其他页面再补会更稳。

| 优先级 | 页面 | 当前理由 |
|---|---|---|
| **P0** | 首页、计划页、记录页 | 决定主任务链是否成立 |
| **P0** | 即时求助页 | 决定 AI 是否以正确方式嵌入产品 |
| **P0** | 消息回流逻辑 | 决定推送是否真的转化为行动 |
| **P1** | 周反馈详情、计划历史 | 决定长期复盘体验 |
| **P1** | 儿童档案、通知设置 | 决定个性化与管理体验 |
| **P2** | 帮助中心、边界说明扩展页 | 决定补充说明与教育性内容 |

## 十六、结论

到这一版为止，iOS 端的页面信息架构已经可以被稳定描述为：**以首页、计划、记录三大主任务页为骨架，以即时求助与消息中心作为上下文回流入口，以档案与设置作为低频支撑层。** 这种结构既承接了前面已经稳定的观察模型、7 天微计划与整体服务架构，也避免了把产品误导为资讯工具或聊天工具。

更重要的是，这套页面结构已经把“服务能力如何变成前台页面”说清楚了。首页是状态聚合，计划页是行动组织，记录页是证据输入，即时求助页是场景化 AI 支持，消息中心是回流中枢。这样一来，下一步无论进入低保真原型、字段结构还是接口拆解，都已经有了可以持续沿用的前台骨架。

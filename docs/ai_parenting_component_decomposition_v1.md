# 18—48个月幼儿家长辅助型 AI 产品：关键页面组件拆解 V1

## 一、文档目标

本稿承接《页面信息架构 V1》与《低保真页面原型 V1》，目的是把已经稳定的页面骨架继续往下推一层，从"页面职责"进入"**组件区职责**"。换句话说，本稿要回答的核心问题不再是"这个页面负责什么"，而是 **每个页面由哪些组件区构成、每个组件区的输入输出是什么、组件区之间如何传递状态、哪些组件可以跨页面复用。**

之所以在低保真原型之后进入组件拆解，而不是直接进入高保真设计或代码实现，是因为组件拆解是连接"页面结构"与"工程实现"的中间层。没有这一层，设计师画出的页面容易只停留在视觉排版层；有了这一层，前端工程师可以直接对应到组件树、状态管理和接口消费逻辑。

本轮拆解同时引入 **多平台实现视角**。虽然当前主入口仍然是 iOS 原生应用，但组件拆解的方式将尽量保持跨平台友好，使同一套组件区定义能够在 SwiftUI、Flutter 或 React Native 等不同实现方案中保持结构一致。这不意味着提前选定跨平台框架，而是确保组件边界、状态流与数据消费方式不因平台切换而需要重新设计。

| 本稿解决的问题 | 本稿暂不展开的问题 |
|---|---|
| 明确每个核心页面的组件区集合与职责 | 具体视觉样式、动效与品牌表达 |
| 固定组件区的输入数据、输出动作与状态依赖 | 接口字段级定义与完整 API 契约 |
| 标注可跨页面复用的通用组件 | 组件库代码实现与构建配置 |
| 结合多平台视角说明组件设计约束 | 特定框架的 Widget / View 命名 |
| 为后续数据结构设计提供前台消费清单 | 数据库表结构与 ORM 细节 |

## 二、组件拆解的设计原则

在进入具体页面之前，需要先固定几条贯穿全局的组件设计原则。这些原则不是为了约束视觉表达，而是为了确保拆出来的组件能够同时满足"设计可评审""工程可实现""多端可复用"三个要求。

**第一，组件区以数据消费边界划分，而不是以视觉区块划分。** 也就是说，两个视觉上紧挨着的卡片，如果消费的数据来源不同、状态生命周期不同，就应该被拆成独立组件区。反过来，如果两块区域消费同一份数据且状态联动，则可以归入同一组件区。

**第二，每个组件区都有明确的输入、输出和交互契约。** 输入是组件区需要外部提供的数据或状态；输出是组件区对外触发的动作（导航、回写、刷新等）；交互契约是组件区内部处理用户操作的方式。这样定义之后，组件区就可以被独立开发、独立测试和跨页面移植。

**第三，状态管理分为页面级与全局级。** 页面级状态只在当前页面生命周期内有效，例如"当前选中的日任务是哪天"；全局级状态跨页面持续存在且需要回写后端，例如"任务完成状态""最新记录条目"。组件区必须明确自己依赖的是哪一级状态。

**第四，跨平台友好意味着组件边界不依赖平台特有能力。** 组件区的定义应该用"状态卡""动作按钮组""输入表单区""时间线列表"这类语义化名称，而不是"UICollectionView""ListView""Column"这类平台绑定名称。具体到 SwiftUI、Flutter 或 React Native 时再做映射。

**第五，交互密度为移动端优先设计。** 所有组件区的信息密度、点击热区和滚动结构都以 iPhone 竖屏为基准。如果后续需要适配 iPad 或 Android 大屏，通过组件区的排列方式（单列变双列、卡片变侧边栏等）调整，而不是重新设计组件内部结构。

| 设计原则 | 对组件拆解的具体要求 |
|---|---|
| **数据消费边界划分** | 同一数据源的内容归入同一组件区 |
| **明确输入输出契约** | 每个组件区列出需要的数据、触发的动作 |
| **状态分级管理** | 区分页面级临时状态与全局级持久状态 |
| **跨平台语义化命名** | 不绑定特定平台 Widget 名称 |
| **移动端密度优先** | iPhone 竖屏为基准，大屏通过布局调整适配 |

## 三、通用组件定义

在进入各页面拆解之前，先把出现频率最高、可以跨页面直接复用的通用组件固定下来。这样做的好处是：后续每个页面的组件区可以直接引用通用组件，而不是每次都重新定义。

在本文档中，**"回写"统一指"更新全局状态并触发后端持久化"**。如果某个操作仅影响页面级临时状态而不需要后端持久化，会单独标注为"页面级写入"。

### 3.1 状态标签组（StatusTagGroup）

一组可以横向排列的标签，用于展示当前阶段、主题、风险层级等分类信息。标签数量动态，样式分为强调（深底白字）和普通（浅底深字）两种。本组件仅用于 **纯展示** 场景。当标签需要支持选中交互时，应使用 3.9 中定义的 SelectableTagGroup。

| 属性 | 说明 |
|---|---|
| **输入** | 标签数组（每项含文本、样式类型） |
| **输出** | 标签点击事件（可选，仅用于跳转，不用于选中） |
| **状态依赖** | 无内部状态 |
| **跨平台说明** | 各平台均需自定义封装 Tag/Chip 组件，原生组件库通常不直接提供 |

### 3.2 动作按钮组（ActionButtonGroup）

一组水平排列的按钮，通常包含一个主要按钮和零至两个次要按钮。主要按钮视觉突出，次要按钮保持克制。

| 属性 | 说明 |
|---|---|
| **输入** | 按钮数组（每项含文本、样式级别、目标动作） |
| **输出** | 按钮点击事件（导航、提交、回写等） |
| **状态依赖** | 可选的加载中 / 禁用状态 |
| **跨平台说明** | 各平台使用原生按钮组件横向排列即可，无需特殊封装 |

### 3.3 信息摘要卡（SummaryCard）

一张包含标题、副标题、正文摘要和可选动作按钮的卡片。用于在各页面中展示单一主题的核心信息。

| 属性 | 说明 |
|---|---|
| **输入** | 标题、副标题、正文、可选标签组、可选动作按钮组 |
| **输出** | 卡片点击（进入详情）或按钮点击（执行动作） |
| **状态依赖** | 可选的未读 / 已完成标记 |
| **跨平台映射** | SwiftUI: VStack + Card / Flutter: Card + Column / RN: View + Shadow |

### 3.4 双列信息区（SplitInfoPanel）

左右两列信息展示，每列包含标题和简短描述。适合展示并列的两类信息，例如"主练习"与"自然嵌入"。

| 属性 | 说明 |
|---|---|
| **输入** | 左列标题与内容、右列标题与内容 |
| **输出** | 可选的列点击事件 |
| **状态依赖** | 无内部状态 |
| **跨平台映射** | SwiftUI: HStack + VStack / Flutter: Row + Expanded / RN: View + flex |

### 3.5 时间线列表（TimelineList）

按时间倒序排列的条目列表，每条包含时间标记、标题和简短描述。适合记录回看、消息列表等场景。

| 属性 | 说明 |
|---|---|
| **输入** | 条目数组（每项含时间、标题、描述、可选目标页） |
| **输出** | 条目点击事件（导航到详情或目标页） |
| **状态依赖** | 可选的分页加载状态 |
| **跨平台映射** | SwiftUI: List + ForEach / Flutter: ListView.builder / RN: FlatList |

### 3.6 底部导航栏（BottomTabBar）

固定在屏幕底部的三 Tab 导航栏：首页、计划、记录。当前激活项高亮显示。

| 属性 | 说明 |
|---|---|
| **输入** | Tab 数组（每项含图标、文本、目标页）、当前激活 Tab |
| **输出** | Tab 切换事件 |
| **状态依赖** | 全局级：当前激活 Tab 索引 |
| **跨平台映射** | SwiftUI: TabView / Flutter: BottomNavigationBar / RN: Tab.Navigator |

### 3.7 顶部工具栏（TopToolBar）

页面顶部区域，包含页面标题、副标题和右侧图标按钮组（如消息入口、档案入口）。

| 属性 | 说明 |
|---|---|
| **输入** | 标题、副标题、右侧图标按钮数组（含未读徽标数） |
| **输出** | 图标按钮点击事件 |
| **状态依赖** | 全局级：未读消息数 |
| **跨平台映射** | SwiftUI: NavigationBar / Flutter: AppBar / RN: Header |

### 3.8 输入表单区（InputFormSection）

支持文本输入、标签选择或语音录入的表单区域。根据场景动态组合不同输入类型。

| 属性 | 说明 |
|---|---|
| **输入** | 输入项数组（每项含类型、标签、占位文本、可选项） |
| **输出** | 值变更事件、提交事件 |
| **状态依赖** | 页面级：当前输入值、校验状态 |
| **跨平台说明** | 各平台使用原生表单组件，需处理键盘弹出时的滚动适配 |

### 3.9 可选中标签组（SelectableTagGroup）

与 StatusTagGroup 外观相似，但支持单选或多选交互。选中后标签切换为强调样式，并通过回调通知父组件当前选中集合。本组件用于所有需要用户选择的场景，例如记录页快速打点（多选）、周反馈页下周决策（单选）、完成面板完成状态（单选）。

| 属性 | 说明 |
|---|---|
| **输入** | 标签数组（每项含 ID、文本、是否默认选中）、选择模式（single / multiple） |
| **输出** | 选中变更事件（返回当前选中 ID 集合） |
| **状态依赖** | 页面级：当前选中集合 |
| **跨平台说明** | 各平台需自定义封装，SwiftUI 可基于 Toggle 或自定义 Button，Flutter 可基于 FilterChip / ChoiceChip |

## 四、首页组件拆解

首页在页面信息架构中被定义为 **状态判断页**：让家长快速知道当前阶段、今天先做什么、是否有需要处理的回流。因此首页的组件拆解必须围绕"判断—跳转"而非"沉浸—消费"来组织。

### 4.1 组件区集合

首页从上到下由以下组件区构成：

| 序号 | 组件区名称 | 职责定义 |
|---|---|---|
| H-1 | **TopToolBar** | 展示儿童昵称与月龄，提供消息入口与档案入口 |
| H-2 | **FocusCard（当前周焦点卡）** | 展示本周阶段、主题、状态级别与周焦点摘要 |
| H-3 | **TodayTaskCard（今日任务卡）** | 展示今天最优先的 1—2 个动作，提供"去执行"跳转 |
| H-4 | **RecentRecordSummary（最近记录摘要）** | 展示最近记录条数与摘要，告知是否影响计划理解 |
| H-5 | **PendingReturnCard（待处理回流卡）** | 展示最需要处理的 1—2 条回流事项，提供深链跳转 |
| H-6 | **QuickHelpEntry（即时求助入口）** | 提供"现在怎么办"快捷入口，携带当前上下文 |
| H-7 | **BottomTabBar** | 底部三 Tab 导航，当前高亮"首页" |

### 4.2 各组件区详细定义

**H-2 FocusCard（当前周焦点卡）**

这是首页视觉权重最大的区域，承担"一眼看到本周核心判断"的职责。它消费的数据来自当前活跃微计划与儿童档案聚合。

| 属性 | 说明 |
|---|---|
| **输入数据** | 当前阶段名称、本周主题名称、状态级别（正常波动 / 重点关注 / 建议咨询）、焦点摘要文本 |
| **包含子组件** | StatusTagGroup（阶段 + 主题 + 状态级别标签）、正文摘要段、ActionButtonGroup（查看本周计划 / 查看阶段说明） |
| **触发动作** | "查看本周计划"→ 导航到计划页；"查看阶段说明"→ 导航到档案阶段详情页 |
| **状态依赖** | 全局级：当前活跃计划 ID、儿童档案阶段 |
| **空态处理** | 无活跃计划时，切换为"建议开始一份计划"引导卡 |
| **跨平台约束** | 卡片高度自适应内容，标签组横向滚动超出时允许省略 |

**H-3 TodayTaskCard（今日任务卡）**

展示今天最值得执行的动作，但不展开完整任务细节，避免首页承载执行功能。

| 属性 | 说明 |
|---|---|
| **输入数据** | 今日主练习标题与摘要、今日自然嵌入标题与摘要、完成状态 |
| **包含子组件** | SplitInfoPanel（主练习 / 自然嵌入）、ActionButtonGroup（去执行） |
| **触发动作** | "去执行"→ 导航到计划页当日任务详情 |
| **状态依赖** | 全局级：今日任务完成状态 |
| **空态处理** | 今日无任务时不显示该组件区 |
| **跨平台约束** | 双列面板在窄屏时可回退为单列堆叠 |

**H-4 RecentRecordSummary（最近记录摘要）**

告知家长最近记录的状态，强化"记录会影响后续建议"的产品认知。

| 属性 | 说明 |
|---|---|
| **输入数据** | 最近记录条数、最新记录摘要文本、是否影响当前计划理解 |
| **包含子组件** | SummaryCard（含标签"已回写""影响本周判断"） |
| **触发动作** | 卡片点击 → 导航到记录页 |
| **状态依赖** | 全局级：最近记录列表（最多取最新 1—2 条） |
| **空态处理** | 无记录时显示"先补一条记录"引导 |

**H-5 PendingReturnCard（待处理回流卡）**

只展示最需要处理的事项摘要，不替代完整消息中心。

| 属性 | 说明 |
|---|---|
| **输入数据** | 待处理事项数组（每项含类型、摘要、目标页深链） |
| **包含子组件** | SummaryCard（含最重要的 1—2 条摘要）、ActionButtonGroup（查看周反馈 / 打开消息中心） |
| **触发动作** | "查看周反馈"→ 导航到周反馈详情；"打开消息中心"→ 导航到消息列表 |
| **状态依赖** | 全局级：未处理消息列表 |
| **空态处理** | 无待处理事项时隐藏该组件区 |

**H-6 QuickHelpEntry（即时求助入口）**

保持高可达但不升级为主 Tab 的即时求助入口。

| 属性 | 说明 |
|---|---|
| **输入数据** | 当前上下文摘要（阶段 + 计划 + 最近记录关键词） |
| **包含子组件** | SummaryCard（含简短说明）、ActionButtonGroup（现在怎么办？） |
| **触发动作** | "现在怎么办"→ 导航到即时求助页，携带当前上下文 |
| **状态依赖** | 全局级：当前计划 + 最近记录 |

### 4.3 首页组件区之间的状态流

首页各组件区之间不存在直接数据依赖——它们都从全局状态读取数据并独立渲染。这意味着首页适合采用 **响应式数据驱动** 的方式实现：全局状态（计划、记录、消息）发生变化时，各组件区自动更新。

| 状态变化来源 | 影响的首页组件区 |
|---|---|
| 计划状态更新 | FocusCard、TodayTaskCard |
| 新记录提交 | RecentRecordSummary、FocusCard（如影响状态级别） |
| 消息状态变化 | PendingReturnCard、TopToolBar（未读数） |
| 周反馈生成 | PendingReturnCard |

## 五、计划页组件拆解

计划页在页面信息架构中被定义为 **完整行动页**：把"这一周具体怎么做"讲清楚，让家长在同一页面体系中看到动作、话术、观察点和完成入口。

### 5.1 组件区集合

| 序号 | 组件区名称 | 职责定义 |
|---|---|---|
| P-1 | **TopToolBar** | 展示"7 天微计划"标题与周进度指示 |
| P-2 | **WeekOverviewCard（本周主线卡）** | 展示本周主目标、优先场景与焦点 |
| P-3 | **DaySelector（日期选择器）** | 允许家长在 Day 1—Day 7 之间切换查看 |
| P-4 | **DailyTaskPanel（当日任务面板）** | 展示选中日的主练习、自然嵌入、示范话术与观察点 |
| P-5 | **CompletionPanel（完成与复盘面板）** | 提供完成状态选择与"完成后去记录"桥接 |
| P-6 | **WeekExtensionEntry（周反馈与历史入口）** | 提供周反馈详情和计划历史的入口 |
| P-7 | **BottomTabBar** | 底部三 Tab 导航，当前高亮"计划" |

### 5.2 各组件区详细定义

**P-2 WeekOverviewCard（本周主线卡）**

本周计划的顶层信息锚定区，让家长不用向下翻就能回忆本周方向。

| 属性 | 说明 |
|---|---|
| **输入数据** | 计划标题、主目标文本、优先场景数组、当前天数（Day N / 7） |
| **包含子组件** | StatusTagGroup（优先场景标签）、进度指示器（Day N / 7） |
| **触发动作** | 无直接导航，锚定展示 |
| **状态依赖** | 全局级：当前活跃计划对象 |
| **空态处理** | 无活跃计划时显示"开始新计划"引导 |

**P-3 DaySelector（日期选择器）**

一排横向可滑动的日期按钮，允许在七天之间快速切换。当前选中日高亮，已完成日有完成标记。

| 属性 | 说明 |
|---|---|
| **输入数据** | 天数数组（Day 1—Day 7）、每天完成状态、当前选中日 |
| **包含子组件** | 横向按钮组（每个按钮含日期标号与状态图标） |
| **触发动作** | 选中日变更 → 更新 DailyTaskPanel 内容 |
| **状态依赖** | 页面级：当前选中日索引；全局级：各天完成状态 |
| **跨平台约束** | 横向滑动交互在所有移动端框架中均有原生支持 |

**P-4 DailyTaskPanel（当日任务面板）**

计划页的核心执行区，承接选中日的完整任务内容。

| 属性 | 说明 |
|---|---|
| **输入数据** | 主练习标题与说明、自然嵌入标题与说明、示范话术文本、观察点文本 |
| **包含子组件** | SplitInfoPanel（主练习 / 自然嵌入）、SplitInfoPanel（示范话术 / 观察点） |
| **触发动作** | 无直接导航，向下衔接 CompletionPanel |
| **状态依赖** | 页面级：DaySelector 的当前选中日 |
| **跨平台约束** | 双列面板在 iPhone SE 等窄屏设备上自动回退为单列 |

**P-5 CompletionPanel（完成与复盘面板）**

把执行结束和下一步动作放在一起，形成计划 → 记录的关键桥梁。

| 属性 | 说明 |
|---|---|
| **输入数据** | 当日完成选项数组（已执行 / 部分顺利 / 需要补记）、当前选中状态 |
| **包含子组件** | SelectableTagGroup（单选模式，完成选项标签）、ActionButtonGroup（完成后去记录 / 现在求助） |
| **触发动作** | 选中完成状态 → 回写当日完成状态（全局级）；"完成后去记录"→ 导航到记录页并携带当日任务上下文；"现在求助"→ 导航到即时求助页 |
| **状态依赖** | 全局级：当日任务完成状态 |
| **重要性说明** | 这是连接计划页与记录页的最关键桥梁组件，必须在所有平台上保证高可达 |

**P-6 WeekExtensionEntry（周反馈与历史入口）**

连接计划页与其延展页面的入口区。

| 属性 | 说明 |
|---|---|
| **输入数据** | 是否有已生成的周反馈、历史计划数量 |
| **包含子组件** | ActionButtonGroup（查看本周反馈 / 查看历次计划） |
| **触发动作** | 导航到周反馈详情页 / 计划历史列表页 |
| **状态依赖** | 全局级：当前计划是否有周反馈 |

### 5.3 计划页组件区之间的状态流

计划页内部存在明确的数据联动关系：DaySelector 的选择直接驱动 DailyTaskPanel 和 CompletionPanel 的内容切换。

| 交互事件 | 状态变化 | 影响的组件区 |
|---|---|---|
| DaySelector 切换到 Day N | 页面级：selectedDay = N | DailyTaskPanel、CompletionPanel |
| CompletionPanel 选中"已执行" | 全局级：Day N 完成状态 = completed | DaySelector（显示完成标记）、首页 TodayTaskCard |
| "完成后去记录"点击 | 导航参数：携带当日任务主题与场景 | 记录页：快速打点区预填选项 |

## 六、记录页组件拆解

记录页在页面信息架构中被定义为 **证据输入页**：用最低负担捕捉"刚刚发生了什么"，再让系统在后台做结构化归档。

### 6.1 组件区集合

| 序号 | 组件区名称 | 职责定义 |
|---|---|---|
| R-1 | **TopToolBar** | 展示"记录"标题与语音录入快捷入口 |
| R-2 | **QuickCheckPanel（快速打点区）** | 提供最快速的勾选式记录入口 |
| R-3 | **EventRecordForm（关键事件记录表单）** | 提供一句话事件记录与场景标签 |
| R-4 | **VoiceInputSection（语音补充区）** | 提供语音录入与文字切换入口 |
| R-5 | **RecordTimeline（记录时间线）** | 展示最近记录列表，支持回看与筛选 |
| R-6 | **BottomTabBar** | 底部三 Tab 导航，当前高亮"记录" |

### 6.2 各组件区详细定义

**R-2 QuickCheckPanel（快速打点区）**

记录页的最快入口，目标是让家长在 10 秒内留下一条轻量记录。

| 属性 | 说明 |
|---|---|
| **输入数据** | 打点选项数组（由后端基于当前计划的观察点与关注主题动态生成并随计划数据一并返回，如"今天完成了计划动作""哭闹缩短一些""仍然很难转场"；无活跃计划时后端返回通用候选列表） |
| **包含子组件** | SelectableTagGroup（多选模式）、ActionButtonGroup（一键提交） |
| **触发动作** | "一键提交"→ 提交选中标签为新记录条目（全局级回写）→ 刷新 RecordTimeline |
| **状态依赖** | 全局级：当前计划的关注主题与观察点（用于动态生成打点选项）；页面级：已选中的标签集合 |
| **空态处理** | 无活跃计划时使用通用打点选项 |
| **跨平台约束** | 标签组需要支持多选交互，SwiftUI 用 Toggle 实现，Flutter 用 FilterChip |

**R-3 EventRecordForm（关键事件记录表单）**

允许家长用一句话描述发生了什么，并附加场景标签。

| 属性 | 说明 |
|---|---|
| **输入数据** | 场景标签候选列表（换衣、出门、吃饭、见人等）、时间标签候选列表（早上、午后、晚饭后等）、主题标签候选列表 |
| **包含子组件** | InputFormSection（文本输入 + tag_select 类型字段承载场景/时间/主题标签的单选交互） |
| **触发动作** | 提交 → 创建新的关键事件记录条目（全局级回写） |
| **状态依赖** | 页面级：当前输入文本、已选标签 |
| **跨平台约束** | 文本输入区需要处理键盘弹出时的滚动适配 |

**R-4 VoiceInputSection（语音补充区）**

为高压家长提供低负担的复杂情境记录方式。

| 属性 | 说明 |
|---|---|
| **输入数据** | 无预设数据 |
| **包含子组件** | ActionButtonGroup（开始录音 / 改为文字）、录音状态指示器（录音中 / 转写中 / 完成） |
| **触发动作** | 开始录音 → 调用系统麦克风权限；录音完成 → 上传后端转写；转写完成 → 自动填入事件记录或生成独立记录条目 |
| **状态依赖** | 页面级：录音状态（空闲 / 录音中 / 转写中 / 完成） |
| **跨平台约束** | 麦克风权限需要各平台原生处理（iOS AVAudioSession / Android MediaRecorder / Flutter sound_recorder） |
| **权限说明** | 首次使用时触发麦克风权限申请，需要提前展示用途说明 |

**R-5 RecordTimeline（记录时间线）**

让家长看到"我的记录确实在积累"的累积价值展示区。

| 属性 | 说明 |
|---|---|
| **输入数据** | 记录条目数组（每项含时间、标题、描述、是否已同步到计划页） |
| **包含子组件** | TimelineList（通用组件复用） |
| **触发动作** | 条目点击 → 进入记录详情或编辑页 |
| **状态依赖** | 全局级：记录列表（按时间倒序） |
| **分页策略** | 初始加载最近 7 天记录，向上滚动触发分页加载 |

### 6.3 记录页组件区之间的状态流

| 交互事件 | 状态变化 | 影响的组件区 |
|---|---|---|
| QuickCheckPanel 提交 | 全局级：新增记录条目 | RecordTimeline 顶部插入新条目 |
| EventRecordForm 提交 | 全局级：新增记录条目 | RecordTimeline 顶部插入新条目 |
| VoiceInputSection 转写完成 | 全局级：新增记录条目 | RecordTimeline 顶部插入新条目 |
| 来自计划页的携带参数 | 页面级：预填 QuickCheckPanel 选项 | QuickCheckPanel 预选相关标签 |

## 七、即时求助页组件拆解

即时求助页在页面信息架构中被定义为 **场景化支持页**：不是无边界聊天，而是"从问题到动作"的缩短器。在导航层级上，**即时求助页不属于底部三 Tab 体系，而是以全屏 modal 方式从任意页面呼出**。用户可以通过左上角关闭按钮或系统返回手势关闭 modal，返回到呼出前的原页面。这意味着即时求助页没有 BottomTabBar，且不会改变底部 Tab 的当前选中状态。

### 7.1 组件区集合

| 序号 | 组件区名称 | 职责定义 |
|---|---|---|
| A-1 | **TopToolBar** | 展示"现在怎么办"标题与上下文标记 |
| A-2 | **ScenarioSelector（问题场景选择区）** | 提供常见问题场景快捷入口与自由输入 |
| A-3 | **ContextCard（上下文透明卡）** | 展示系统已引用的阶段、计划与最近记录 |
| A-4 | **ThreeStepResultCard（三步支持结果卡）** | 展示"先说什么 / 接着做什么 / 没接住怎么办" |
| A-5 | **FollowUpActionBar（后续动作区）** | 提供"补记为记录""加入本周关注""查看咨询准备建议" |
| A-6 | **BoundaryNote（边界说明区）** | 展示非诊断化提示与咨询升级条件 |

### 7.2 各组件区详细定义

**A-2 ScenarioSelector（问题场景选择区）**

优先提供常见问题场景入口，降低家长在高压时刻的输入负担。

| 属性 | 说明 |
|---|---|
| **输入数据** | 常见场景标签数组（转场崩溃、不肯开口说、见人就躲、总是抢回合等）、自由输入框 |
| **包含子组件** | InputFormSection（tag_select 类型承载场景标签单选 + textarea 类型承载自由输入，折叠态） |
| **触发动作** | 选择场景或提交输入 → 触发 AI 请求，携带上下文 |
| **状态依赖** | 页面级：已选场景或输入文本；全局级：当前儿童阶段与计划（用于上下文） |
| **跨平台约束** | 场景标签需要适配不同屏幕宽度的换行行为 |

**A-3 ContextCard（上下文透明卡）**

显式展示系统引用了哪些上下文，增强家长对 AI 建议的信任。

| 属性 | 说明 |
|---|---|
| **输入数据** | 儿童月龄、当前计划主题、最近记录关键词（最多 3 个） |
| **包含子组件** | StatusTagGroup（上下文标签） |
| **触发动作** | 无直接动作，仅信息展示 |
| **状态依赖** | 全局级：儿童档案、当前计划、最近记录 |
| **设计意图** | 这个组件区是本产品区别于普通聊天 AI 的关键标识——让家长知道 AI 不是凭空回答 |

**A-4 ThreeStepResultCard（三步支持结果卡）**

把 AI 输出压缩成三段式结构化结果，匹配高压时刻的阅读心智。

| 属性 | 说明 |
|---|---|
| **输入数据** | 第一步（先说什么）文本、第二步（接着做什么）文本、第三步（没接住怎么办）文本 |
| **包含子组件** | 三个 SummaryCard（每个承载一步的标题与正文，可选附带示范话术） |
| **触发动作** | 无直接动作 |
| **状态依赖** | 页面级：AI 请求结果（加载中 / 已返回 / 失败） |
| **空态处理** | 请求中显示骨架屏；失败时显示保守的模板化建议 |
| **跨平台约束** | 三步结构在窄屏时改为纵向堆叠 |

**A-5 FollowUpActionBar（后续动作区）**

确保求助结果不悬空，必须回流到系统状态中。

| 属性 | 说明 |
|---|---|
| **输入数据** | 可选动作数组（补记为记录、加入本周关注、查看咨询准备建议） |
| **包含子组件** | ActionButtonGroup |
| **触发动作** | "补记为记录"→ 导航到记录页并预填内容；"加入本周关注"→ 回写下周计划上下文；"查看咨询准备建议"→ 导航到咨询准备页 |
| **状态依赖** | 页面级：当前 AI 会话结果 |
| **重要性说明** | 三个回流动作是避免求助结果悬空的核心设计，必须在所有平台上保证可见 |

**A-6 BoundaryNote（边界说明区）**

始终保持非诊断化原则的页面级提示。

| 属性 | 说明 |
|---|---|
| **输入数据** | 固定文案（"本建议不替代专业评估，如持续困难请考虑咨询"类表达） |
| **包含子组件** | 静态文本块 |
| **触发动作** | 可选：点击展开更详细的边界说明 |

### 7.3 即时求助页组件区之间的状态流

即时求助页的组件区之间存在明确的串行依赖关系：场景选择 → AI 请求 → 结果展示 → 后续动作。

| 交互阶段 | 可见组件区 | 状态 |
|---|---|---|
| 初始态 | ScenarioSelector + ContextCard | 等待用户选择场景 |
| 请求中 | ScenarioSelector + ContextCard + ThreeStepResultCard（骨架屏） | AI 请求加载中 |
| 结果返回 | 全部组件区可见 | 展示完整结果与后续动作 |
| 请求失败 | ScenarioSelector + ContextCard + ThreeStepResultCard（降级模板） + FollowUpActionBar | 展示保守建议 |

## 八、消息中心组件拆解

消息中心在页面信息架构中被定义为 **回流中枢**：不堆通知数量，而是把系统触达翻译成可执行的回流动作。

### 8.1 组件区集合

| 序号 | 组件区名称 | 职责定义 |
|---|---|---|
| M-1 | **TopToolBar** | 展示"消息中心"标题与未读计数 |
| M-2 | **MessageList（消息列表）** | 按回流目标组织的消息条目列表 |
| M-3 | **MessageDetail（消息详情预览）** | 展示选中消息的详情与深链跳转按钮 |
| M-4 | **ProcessingStatus（处理状态概览）** | 弱化展示未读、已深链、待补记等统计 |

### 8.2 各组件区详细定义

**M-2 MessageList（消息列表）**

| 属性 | 说明 |
|---|---|

**M-3 MessageDetail（消息详情预览）**

| 属性 | 说明 |
|---|---|

| **状态依赖** | 页面级：当前选中消息 |

## 九、周反馈页组件拆解

周反馈页在页面信息架构中被定义为 **计划链路延展页**：让"记录影响建议"在页面上真正可见。在导航层级上，**周反馈页以 push 方式从计划页、首页或消息中心进入**，返回时回到呼出前的来源页。它同样不包含 BottomTabBar，而是在顶部提供标准返回按钮。

### 9.1 组件区集合

| F-3 | **OpportunityCard（仍需机会卡）** | 展示仍需多给机会的方向 |
| F-4 | **NextWeekDecisionPanel（下周决策面板）** | 提供继续 / 降难度 / 换焦点等决策选项 |
| F-5 | **ConservativePathNote（更保守路径说明）** | 展示升级条件与咨询准备入口 |

### 9.2 各组件区详细定义

**F-2 PositiveChangeCard 与 F-3 OpportunityCard**

两者结构相同，只是内容方向不同，可以复用同一个组件模板。

| 属性 | 说明 |
|---|---|
| **输入数据** | 标题、正文描述 |
| **包含子组件** | SummaryCard |
| **触发动作** | 无直接动作 |
| **状态依赖** | 全局级：当前周反馈对象 |

**F-4 NextWeekDecisionPanel（下周决策面板）**

| 属性 | 说明 |
|---|---|
| **输入数据** | 决策选项数组（继续同焦点 / 降低难度 / 换焦点） |
| **包含子组件** | SelectableTagGroup（单选模式，决策选项标签）、ActionButtonGroup（确认继续 / 回看计划与记录） |
| **触发动作** | 选中决策 → 回写下周计划方向（全局级）；"回看计划与记录"→ 导航到计划历史 |
| **状态依赖** | 全局级：当前周反馈、计划状态 |

**F-5 ConservativePathNote（更保守路径说明）**

| 属性 | 说明 |
|---|---|
| **输入数据** | 升级条件文本、咨询准备入口 |
| **包含子组件** | SummaryCard（虚线边框样式）、ActionButtonGroup（查看咨询准备建议） |
| **触发动作** | 导航到咨询准备页 |
| **设计意图** | 始终让家长看到"不是只能继续加码训练，还有更保守的合理选择" |

## 十、跨页面可复用组件汇总

基于以上拆解，以下组件在多个页面中被直接复用：

| 通用组件 | 复用页面 | 复用次数 |
|---|---|---|
| **StatusTagGroup** | 首页、计划、即时求助、消息 | 4（H-2, P-2, A-3, M-4） |
| **SelectableTagGroup** | 记录、计划、周反馈 | 4（R-2, P-3, P-5, F-4） |
| **ActionButtonGroup** | 所有页面 | 12+（几乎所有卡片和面板底部） |
| **SummaryCard** | 首页、即时求助、消息、周反馈、计划 | 14+（H-4, H-5, H-6, P-2, P-4, P-6, A-3, A-4×3, A-6, F-2, F-3, F-5, M-3） |
| **SplitInfoPanel** | 首页、计划 | 2（H-3, P-4） |
| **TimelineList** | 记录、消息 | 2（R-5, M-2） |
| **BottomTabBar** | 首页、计划、记录 | 3（全局唯一实例，H-7, P-7, R-6） |
| **TopToolBar** | 所有页面 | 7（H-1, P-1, R-1, A-1, M-1, F-1 + 档案/设置） |
| **InputFormSection** | 记录、即时求助 | 3（R-3, R-4, A-2） |

这意味着在工程实现时，只需要先把这 9 个通用组件做稳，就能覆盖全部核心页面的绝大部分 UI 需求。

## 十一、全局状态模型建议

基于各页面组件区对状态的依赖分析，建议将全局状态组织为以下几个领域：

| 状态领域 | 包含内容 | 写入来源 | 消费页面 |
|---|---|---|---|
| **ChildProfile** | 儿童月龄、阶段、关注主题 | 进入层 / 档案编辑 | 首页、计划、即时求助 |
| **ActivePlan** | 当前计划对象、日任务、完成状态 | 后端同步 / 计划页操作 | 首页、计划 |
| **Records** | 记录列表、最新记录摘要 | 记录页提交 | 首页、记录、即时求助 |
| **Messages** | 消息列表、未读数、处理状态 | 后端推送 / 消息页操作 | 首页、消息中心 |
| **WeeklyFeedback** | 周反馈内容、查看状态 | 后端生成 | 首页、计划、周反馈 |
| **AISession** | 当前求助会话、结果、上下文 | 即时求助页操作 | 即时求助 |
| **Navigation** | 当前 Tab、页面栈 | 全局导航事件 | BottomTabBar、各页面 |

### 全局状态的跨平台实现映射

| 平台 | 推荐状态管理方式 | 说明 |
|---|---|---|
| **SwiftUI** | @Observable / @EnvironmentObject + Repository 模式 | 各状态领域封装为独立 Repository，通过环境注入 |
| **Flutter** | Riverpod / Bloc | 各状态领域对应独立 Provider 或 Bloc |
| **React Native** | Zustand / Redux Toolkit | 各状态领域对应独立 Store Slice |

## 十二、多平台实现约束汇总

以下约束从组件拆解中提炼，适用于任何平台的实现：

| 约束类别 | 具体要求 | 影响的组件区 |
|---|---|---|
| **屏幕适配** | SplitInfoPanel 在宽度 < 360pt 时回退为单列 | 首页 H-3、计划页 P-4、即时求助 A-4 |
| **键盘适配** | 文本输入区激活时，页面自动滚动到输入框上方 | 记录 R-3、即时求助 A-2 |
| **权限处理** | 麦克风权限需要在录音前主动申请并说明用途 | 记录 R-4 |
| **离线容忍** | 快速打点和事件记录需要在离线时本地暂存，恢复后同步 | 记录 R-2、R-3 |
| **加载状态** | AI 请求需要骨架屏过渡，不允许空白等待 | 即时求助 A-4 |
| **导航一致性** | 底部 Tab 切换不重置页面内滚动位置 | 全局 BottomTabBar |
| **深链支持** | 消息点击需要从通知直接落到目标页面的目标组件区 | 消息 M-2、首页 H-5 |
| **手势安全区** | 底部 Tab 区域需要预留 Safe Area，避免与系统手势冲突 | 全局 BottomTabBar |

## 十三、结论

到这一版为止，核心页面的组件拆解已经从"页面级职责"下沉到了"组件区级职责"。每个组件区都有明确的输入数据、输出动作、状态依赖和跨平台映射。更重要的是，这份拆解不只是给设计师看的线框说明，而是同时可以作为前端工程师的组件开发清单和后端工程师的数据消费清单。

整个组件体系可以被归纳为：**9 个通用组件覆盖全局复用需求，6 个核心页面共计 30+ 个组件区实例，7 个全局状态领域支撑跨页面数据流，12 项多平台实现约束确保跨端结构一致。** 这样一来，下一步无论进入高保真设计、数据结构定义还是前端工程启动，都已经有了可以直接对照的组件级骨架。

| 推荐下一步 | 产出目标 |
|---|---|
| **数据结构草案 V1** | 把全局状态模型映射到后端对象结构，明确 API 消费契约 |
| **高保真设计启动** | 基于组件拆解进入品牌、配色、字体与动效层 |
| **组件库工程初始化** | 先实现 9 个通用组件，再按页面优先级逐步填充 |

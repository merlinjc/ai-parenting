# 18—48个月幼儿家长辅助型 AI 产品：整体服务架构草案 V1

## 一、文档目标

本稿用于在进入页面信息架构之前，先把当前产品的**整体服务形态**固定下来。它承接《产品设计草案》《观测模型结构稿 V1.1》《三阶段具体观察项清单 V1》《7 天微计划模板 V1》与《7 天微计划示范样稿 V1》，进一步回答一个系统层的问题：**这款产品并不只是内容和页面的组合，而是一套由 iOS 客户端、业务后端、AI 编排层与消息触达链路共同组成的连续服务系统。**

这份文档的作用，不是提前进入代码实现，也不是把接口字段和数据库结构一次性锁死，而是为后续页面信息架构、数据结构和生成逻辑提供稳定边界。它要说明四件事：家长主要在什么端完成基础使用；哪些能力必须放在服务端；AI 通过什么路径被接入；消息推送如何形成触达与回流闭环。这样，后续的页面设计就不再只是“把内容放进页面”，而是能准确映射到真实的系统职责与状态流转。

| 本稿解决的问题 | 本稿暂不展开的问题 |
|---|---|
| 明确客户端、后端、AI 与外部服务的职责边界 | 页面视觉稿与交互细节 |
| 固化账户、档案、记录、微计划、AI、推送等核心模块 | 接口字段级定义与数据库表设计 |
| 说明登录、记录、AI 输出、消息触达等关键闭环 | 代码实现与部署方案细节 |
| 对齐非诊断化表达、数据保存边界与状态回流原则 | 多云、多活或复杂基础设施方案 |

## 二、架构定位与设计原则

当前产品的核心定位仍然不变：它是一款**面向家长端的发育观察与 AI 轻干预教练产品**，而不是儿童高频独立使用的陪伴应用，也不是代替专业评估的判断系统。因此，整体服务架构必须服务于一个非常清楚的中心：**帮助家长更稳定地观察、记录、执行、复盘，并在需要时更早进入专业支持路径。**

从服务设计上看，系统需要同时满足三个条件。第一，主入口必须足够轻，让家长可以在 iOS 端完成绝大多数基础动作。第二，核心状态必须被服务端统一托管，避免重要记录只停留在本地或由客户端直接拼接业务逻辑。第三，AI 调用必须受到统一编排和边界控制，不能让模型输出脱离产品既定的非诊断化框架。换句话说，系统的目标不是让 AI 直接“回答更多”，而是让整个产品链路更稳定地支撑“观察—微计划—周反馈—即时求助”这条主路径。

| 设计原则 | 架构层面的直接要求 |
|---|---|
| **家长端主入口** | 基础使用全部优先在 iOS 完成，减少跨端跳转 |
| **服务端统一托管状态** | 登录态、档案、记录、计划、会话摘要与通知状态均由后端管理 |
| **AI 统一经服务端代理** | 不由客户端直连模型，统一控制密钥、上下文、输出结构与审计 |
| **同步与异步分层** | 即时页面动作走同步接口；生成、推送、重试与聚合走异步任务 |
| **低屏幕、强亲子、非诊断化** | AI 输出只服务家长决策与家庭互动，不转向儿童高频屏幕交互 |
| **状态可回流** | 推送送达、点击、计划完成、记录提交都能回写系统状态 |

## 三、整体分层结构

建议将系统先固定为四层结构：**iOS 应用层、业务后端层、AI 编排层、外部服务层。** 这种划分的好处在于，既能清楚表达用户可见界面与系统内部能力的区别，也能把“产品逻辑”“AI 调用逻辑”和“第三方基础能力”分开管理。对当前阶段而言，这种分层比一开始拆成多个独立微服务更稳妥，也更适合文档继续演进。

其中，iOS 应用层负责家长可见、可操作、可感知的一切前台行为；业务后端层负责统一业务状态与数据；AI 编排层负责把模型能力转化为符合产品边界的结构化输出；外部服务层则承接模型供应商、推送通道和补充基础能力。后续即使系统规模扩大，这四层结构也仍然成立，只是内部模块会逐步拆分得更细。

| 层级 | 定位 | 主要职责 |
|---|---|---|
| **iOS 应用层** | 用户主入口 | 登录注册、儿童档案、观察记录、微计划查看、每日打卡、AI 交互入口、消息承接、本地轻缓存 |
| **业务后端层** | 统一业务中台 | 账户鉴权、档案管理、记录存储、计划状态、周反馈聚合、消息调度、审计支持 |
| **AI 编排层** | AI 能力屏障 | Prompt 模板、上下文拼装、模型路由、结果结构化、边界校验、失败降级 |
| **外部服务层** | 第三方依赖 | AI 服务商 OpenAPI、Apple Push Notification service、对象存储、日志监控等 |

## 四、iOS 应用层职责

既然用户已经明确基础使用都在 iOS 上实现，那么端上的设计重点就不是“把所有逻辑都做完”，而是**把所有高频使用路径承接好**。iOS 应用需要承担的，是身份入口、家庭信息承接、日常记录录入、微计划消费、消息接收与即时回流等任务。它应当是用户的稳定入口，但不应成为复杂业务规则的最终判定者。

端上尤其适合承接三类能力。第一类，是与体验直接相关的轻逻辑，例如表单输入、任务展示、离线轻缓存、通知跳转和本地权限管理。第二类，是让用户在低负担下完成记录的交互组织，例如把每日记录拆成勾选、短文本、语音转写入口或快速回顾卡片。第三类，是把后端已经生成好的结构化结果以稳定方式展示给用户，例如本周计划卡、今日任务、周反馈摘要与消息详情。

| iOS 端模块 | 主要职责 | 不建议放在端上的职责 |
|---|---|---|
| **账户入口** | 登录、注册、会话续期、退出登录、设备绑定 | 密钥管理、复杂风控规则 |
| **家庭与儿童档案** | 录入儿童年龄阶段、照护者关系、基础家庭信息 | 风险分层判定逻辑 |
| **观察与记录入口** | 结构化勾选、事件记录、语音上传、周回顾填写 | 趋势聚合与长期判断 |
| **微计划与任务页** | 展示 7 天计划、日任务、完成状态、周末判断 | 独立生成计划与升级判断 |
| **AI 交互入口** | 承接“现在怎么办”等即时求助问题并展示回复 | 直接调用外部模型 OpenAPI |
| **消息中心与通知落地页** | 承接提醒、回流到计划页或记录页、展示送达内容 | 推送策略决策与任务调度 |

## 五、业务后端层：统一业务中台

后端建议先采用**模块化单体**。原因并不复杂：当前产品仍处于结构定义和链路收敛阶段，真正重要的是把状态关系理顺，而不是预先把组织复杂度做高。用一个统一后端承接账户、记录、计划、AI、通知与运营支持，可以更快形成稳定闭环，也便于后续根据访问压力和团队分工再逐步拆分。

在这个统一后端中，至少需要固定六类核心业务模块。它们不是随意罗列的功能块，而是贯穿用户全路径的状态载体。账户模块解决“谁在使用”；档案模块解决“服务对象是谁”；记录模块解决“最近发生了什么”；微计划模块解决“接下来一周怎么支持”；AI 会话与编排模块解决“系统如何理解并生成建议”；通知模块则解决“什么时候把合适的信息送回用户”。

| 后端模块 | 主要对象 | 核心职责 |
|---|---|---|
| **账户与身份模块** | 用户、会话、设备、鉴权令牌 | 注册登录、鉴权、设备绑定、多端会话管理 |
| **家庭与儿童档案模块** | 儿童资料、年龄阶段、照护者关系、基础画像 | 为观察、计划与 AI 输出提供上下文 |
| **观察与记录模块** | 打点记录、关键事件、周回顾、语音转写结果 | 保存证据、聚合趋势、沉淀历史轨迹 |
| **微计划与任务模块** | 7 天计划、日任务、完成状态、周总结去向 | 管理计划版本、任务推进与状态回写 |
| **AI 会话与编排模块** | 问题输入、上下文摘要、模型结果、结构化卡片 | 连接模型输出与业务语义 |
| **通知与消息模块** | 推送任务、消息模板、送达结果、点击回流 | 负责触达与回流闭环 |
| **审计与运营支持模块** | 日志、配置、人工回看、内容审核记录 | 支持质量控制、异常排查与策略调整 |

## 六、AI 编排层：把模型能力转成产品能力

在本产品中，AI 不是一个独立悬浮的聊天功能，而是必须被放进业务上下文里的**生成与解释能力**。因此，AI 编排层的职责不是简单转发用户输入，而是把儿童阶段、近期观察、当前微计划状态、风险层级和家长提问场景组织起来，再交给模型生成结果。随后，系统还要把模型输出重新整理成产品可用的结构，例如计划卡片、回应建议、周反馈摘要或风险升级提示。

这一层必须完全在服务端实现。客户端不直接调用模型，一方面是为了密钥与成本控制，另一方面更重要的是统一质量边界。只有经过编排层，系统才能保证不同模型供应商、不同 Prompt 模板和不同版本策略下，仍然输出同一种产品口径：**克制、可执行、非诊断化、强调真实生活中的家长支持。**

| AI 编排环节 | 主要职责 | 输出形式 |
|---|---|---|
| **Prompt 模板管理** | 固定不同场景下的生成框架与语气边界 | 计划模板、即时求助模板、周反馈模板 |
| **上下文拼装** | 汇总儿童阶段、近期记录、计划状态与家庭场景 | 面向模型的结构化输入 |
| **模型供应商适配** | 统一对接不同 OpenAPI，屏蔽底层差异 | 标准化调用请求与响应 |
| **结果结构化** | 把模型长文本改写成卡片、槽位和状态字段 | 微计划、话术建议、周结论 |
| **边界与风控校验** | 检查是否出现诊断化、过度承诺或不当建议 | 拦截、重试、降级或人工审核 |
| **失败降级处理** | 模型超时、失败或输出不合格时退回保守方案 | 模板化建议、稍后再试、仅返回记录摘要 |

## 七、外部服务层与推送链路

系统对外部服务的依赖应保持克制，但不能模糊。当前阶段，至少有两类外部能力是明确需要的：第一类是**AI 服务商 OpenAPI**，用于承接生成型能力；第二类是 **Apple Push Notification service（APNs）**，用于将提醒、计划续做、周反馈完成等消息送达 iOS 端。除此之外，还可以按需要接入对象存储、日志监控和消息补充通道，但不宜在本轮文档中展开成复杂平台体系。

推送链路本身不只是“发出去”这么简单。对本产品而言，消息的价值在于它能否把用户重新带回正确的任务入口。因此，推送系统需要记录送达结果、点击结果和回流页面，并把这些信息回写到业务状态中。只有这样，后续才能评估提醒是否真正促进了记录、复盘和计划执行，而不是只形成打扰。

| 外部能力 | 当前角色 | 后端需要承担的工作 |
|---|---|---|
| **AI 服务商 OpenAPI** | 提供文本生成与结构化建议的底层能力 | 统一鉴权、模型路由、重试、成本与版本控制 |
| **APNs** | 向 iOS 设备发送系统通知 | 消息模板、发送时机、送达记录、点击回流 |
| **对象存储** | 保存语音、附件或结构化导出结果 | 访问控制、生命周期管理、引用关系 |
| **日志与监控** | 记录错误、延迟、失败与关键链路表现 | 告警、排障、质量回溯 |

## 八、核心数据对象与状态边界

虽然本轮不进入数据库细节，但为了让页面信息架构和后续接口定义有稳定依托，仍然需要先把关键数据对象说清楚。当前系统至少应围绕七类对象组织：**用户、设备、儿童档案、观察记录、微计划、AI 会话、消息通知。** 这七类对象几乎覆盖了产品从首次进入到长期使用的全部关键状态。

其中，最重要的不是对象数量，而是对象之间的连接关系。儿童档案是绝大多数业务逻辑的上下文中心；观察记录是微计划和 AI 交互的重要输入；微计划既会消费记录，也会产生新的完成状态和周反馈；消息通知则是把状态变化重新送回用户的触达机制。这样一来，后续页面层看到的“首页卡片”“今日任务”“消息提醒”其实都能对应回这些核心对象。

| 核心对象 | 关键字段方向 | 与其他对象的关系 |
|---|---|---|
| **用户** | 账号标识、登录方式、照护者角色 | 关联设备、儿童档案、消息偏好 |
| **设备** | 设备标识、推送令牌、平台版本 | 关联用户与通知送达 |
| **儿童档案** | 出生年月、年龄阶段、基础照护信息 | 关联观察、计划、AI 上下文 |
| **观察记录** | 主题、条目、事件、时间、场景、风险层级 | 作为计划与 AI 的主要输入 |
| **微计划** | 焦点、阶段、日任务、完成状态、周结论 | 消费记录并生成新的执行状态 |
| **AI 会话** | 提问场景、上下文摘要、响应结果、版本信息 | 关联计划、记录与消息 |
| **消息通知** | 模板、触发原因、发送结果、点击结果 | 回流到计划页、记录页或求助页 |

## 九、四条核心业务闭环

为了让整体架构不只停留在模块列表层，本稿建议把系统能力收束成四条最重要的业务闭环：**账户闭环、记录闭环、AI 闭环、消息闭环。** 它们分别对应“进来”“留下”“得到建议”“被重新唤回”四个关键问题。这四条闭环共同构成了后续页面信息架构的底层逻辑。

### 1. 账户闭环

账户闭环解决用户首次进入和持续可用的问题。家长在 iOS 完成注册或登录后，后端建立用户身份、设备绑定和会话状态；当会话过期、设备更换或推送令牌变化时，系统也能继续维护这条关系。对当前产品而言，账户体系不是增长工具优先，而是**后续所有观察、计划和消息状态能否稳定归属的前提**。

| 步骤 | 主要动作 | 结果 |
|---|---|---|
| **注册 / 登录** | iOS 提交身份信息，后端完成鉴权 | 建立用户身份与会话 |
| **设备绑定** | 客户端上报设备与推送令牌 | 建立触达能力 |
| **会话续期** | 客户端发起刷新，后端校验令牌 | 保持基础使用连续性 |
| **退出或失效处理** | 后端清理会话与部分设备态 | 避免状态混乱与误推送 |

### 2. 记录闭环

记录闭环是本产品最重要的“证据输入”链路。家长在 iOS 端完成轻量打点、关键事件记录或周回顾后，后端保存结构化信息，并与儿童阶段、主题、场景和时间窗关联。随后，系统可基于这些记录生成趋势理解、微计划输入或周反馈摘要。

这里最重要的架构判断是：**记录先于建议，证据先于生成。** 如果记录没有被稳定保存、聚合和回看，那么 AI 只能不断输出泛化建议，无法形成真正个性化的连续服务。

| 步骤 | 主要动作 | 结果 |
|---|---|---|
| **记录提交** | 家长完成勾选、文本或语音输入 | 形成结构化观察证据 |
| **证据落库** | 后端写入主题、场景、时间与风险相关信息 | 可被检索与聚合 |
| **趋势聚合** | 系统按周或按焦点汇总变化 | 为计划与周反馈提供输入 |
| **状态回写** | 首页、计划页、反馈页显示最新记录状态 | 用户看到“记录确实产生影响” |

### 3. AI 闭环

AI 闭环的关键不在于“发一问一答”，而在于后端能否把业务上下文组织好，再把输出稳妥回写。系统在收到家长提问、周反馈生成请求或计划更新请求后，应先提取必要上下文，再统一调用模型，随后完成结构化整理和状态入库。这样，AI 的每次输出都不是一次孤立文本，而是系统状态的一部分。

| 步骤 | 主要动作 | 结果 |
|---|---|---|
| **请求进入** | 家长提问或系统触发生成任务 | 明确场景类型 |
| **上下文组装** | 汇总儿童阶段、最近记录、当前计划、风险层级 | 形成模型输入 |
| **模型调用** | 后端统一调用 OpenAPI | 获取生成结果 |
| **结果整理** | 结构化为话术、计划卡片、摘要或结论 | 进入产品可消费形态 |
| **状态回写** | 保存到计划、会话或反馈对象 | 支撑后续展示与复盘 |

### 4. 消息闭环

消息闭环解决“系统何时主动回来找用户”的问题。对本产品而言，最有价值的消息并不是泛泛提醒，而是与当前计划节奏、记录状态和周反馈节点紧密相关的触达。例如，提醒家长完成今日任务、提示某份周反馈已经生成、提醒某个高压场景需要补充记录，或在需要时建议查看咨询准备清单。

| 步骤 | 主要动作 | 结果 |
|---|---|---|
| **事件触发** | 计划到日、周反馈完成、记录缺失、重点状态变化 | 形成候选消息 |
| **后端调度** | 选择模板、时间与触达对象 | 生成推送任务 |
| **APNs 发送** | 向 iOS 设备推送通知 | 用户收到消息 |
| **点击回流** | 用户进入对应页面 | 形成行为回写 |
| **效果记录** | 记录送达、打开与转化 | 支撑后续提醒优化 |

## 十、同步接口、异步任务与状态回流

从系统运行方式看，本产品需要清楚区分哪些动作应立即返回，哪些动作更适合后台完成。登录注册、档案编辑、记录提交、计划查询、消息列表获取等，通常应通过同步接口完成，因为用户需要立即看到结果。相反，周反馈生成、较长的 AI 总结、推送调度、失败重试和周期聚合，则更适合放到异步任务体系中处理。

这种分层有两个好处。第一，它能避免把外部模型延迟直接暴露给端上高频页面。第二，它能让系统在失败时保留更好的恢复能力，例如重试、延后发送、返回保守结果或标记为稍后更新。对当前产品而言，这一点尤其重要，因为用户真正感知的是“我的记录是否被接住、计划是否稳定可用”，而不是后端是否一次性完成了所有计算。

| 类型 | 适合承载的动作 | 设计重点 |
|---|---|---|
| **同步接口** | 登录、档案编辑、记录提交、计划查看、消息读取 | 响应稳定、状态明确、错误可解释 |
| **异步任务** | AI 长文本生成、周反馈聚合、推送调度、失败重试 | 幂等、可追踪、可恢复 |
| **状态回流** | 推送点击、任务完成、记录更新、反馈查看 | 回写首页、计划页和消息中心状态 |

## 十一、安全边界、隐私与风险控制

由于系统同时涉及儿童相关观察信息、家庭互动记录与外部 AI 调用，安全边界必须在服务架构层就前置写明。首先，**密钥与模型调用权限只保留在服务端**，客户端不得直连模型服务。其次，所有输入都应按最小必要原则保存，避免为了“以后可能有用”而长期存储无边界的家庭隐私。再次，日志与审计中应避免泄露可直接识别家庭身份的原始内容，尤其不宜把完整敏感输入直接写入普通业务日志。

在产品边界上，系统需要继续坚持“非诊断化、非替代性、非高压训练化”的原则。也就是说，即便 AI 能够根据记录给出支持建议，系统也不应输出疾病标签、不应承诺治疗效果，也不应把需要专业判断的问题长期留在家庭端自行解决。当风险状态进入“建议咨询”时，架构上应支持切换到更保守的输出模式，例如稳定支持建议、补充记录提示和咨询准备清单，而不是继续加强训练任务。

| 安全与边界项 | 架构要求 |
|---|---|
| **鉴权与会话安全** | 所有核心写操作均需鉴权，设备与用户关系需可追踪 |
| **AI 密钥管理** | 密钥仅保存在服务端，不下发客户端 |
| **输入校验** | 记录、提问、语音转写结果均需做格式与长度校验 |
| **隐私最小化保存** | 只保存实现服务所需的必要信息，避免无边界囤积 |
| **日志脱敏** | 业务日志与监控日志避免直接暴露家庭隐私 |
| **非诊断化输出** | 对模型输出进行边界校验，必要时拦截或改写 |
| **风险升级切换** | 当进入建议咨询路径时，切换为保守支持与咨询准备模式 |

## 十二、异常降级与可靠性策略

系统真正的可靠，不是“永不失败”，而是在失败时仍然能给出可解释、可恢复、不会误导用户的结果。由于本产品依赖外部模型服务，最常见的不稳定来源往往不是数据库，而是 AI 响应延迟、调用失败或输出不符合结构要求。因此，系统需要在架构上预留降级路径，而不是把每一次模型失败都变成用户端空白。

更稳妥的做法是：当模型超时、供应商异常或输出不通过边界校验时，系统优先退回到模板化、较保守的结果。例如，即时求助场景下可返回固定结构的安抚与下一步建议；周反馈场景下可优先回传记录摘要与待补充信息；微计划生成场景下可回退到统一模板框架而非完全不可用。这样做虽然牺牲了一部分“智能感”，但能显著提高整体服务的连续性与可信度。

| 异常场景 | 建议降级方式 | 用户感知目标 |
|---|---|---|
| **模型超时** | 返回“稍后补全”的保守结果，并保留任务重试 | 不让页面长时间卡死 |
| **模型输出不合格** | 触发结构化重试或回退到模板化建议 | 不把不合格内容直接展示给家长 |
| **推送发送失败** | 记录失败原因并重试，必要时回到站内消息 | 保证重要提醒不过度丢失 |
| **异步任务中断** | 基于幂等键恢复任务或重新入队 | 降低重复执行与状态混乱 |
| **部分状态回写失败** | 保留补偿机制，稍后同步端上状态 | 避免首页与详情页状态不一致 |

## 十三、建议的实现形态与演进路径

虽然本稿不进入代码实现，但为了避免后续文档与实现完全脱节，仍然可以给出一个克制的建议方向。客户端建议采用**iOS 原生应用**作为主形态，界面层可优先考虑 SwiftUI；后端建议采用**统一 API 服务 + 模块化单体**；数据存储可围绕关系型数据库组织核心业务对象；异步任务可承接推送、AI 重试与周期聚合；外部依赖则以 AI OpenAPI 与 APNs 为当前最关键两项。

这一路径的重点不在于技术新颖，而在于适配当前产品阶段。先用单一后端把账户、档案、记录、计划、AI 和通知这几条主链路打通，等到调用量、团队分工或运维要求明显提升后，再考虑按模块拆分。这样做能够最大程度避免“架构先行、业务未稳”的技术债。

| 方向 | 当前建议 | 说明 |
|---|---|---|
| **客户端** | iOS 原生应用 | 承接核心使用路径与通知回流 |
| **界面层** | SwiftUI 优先 | 适合快速迭代家长端信息结构 |
| **服务端** | 模块化单体 | 有利于当前阶段快速收敛产品闭环 |
| **核心存储** | 关系型数据库优先 | 适合账户、档案、记录、计划等强状态对象 |
| **异步处理** | 队列 + 任务调度 | 用于推送、AI 重试、周期聚合 |
| **外部依赖** | AI OpenAPI + APNs | 当前最关键两类第三方能力 |

## 十四、对后续页面信息架构的直接支撑

这份服务架构文档的价值，不只是给后端画边界，更重要的是它为页面信息架构提供了**稳定映射关系**。后续当我们设计首页、计划页、记录页、消息中心和即时求助入口时，就不必再从零思考“这个页面背后由谁提供数据、何时回写状态、哪些内容是 AI 生成、哪些内容是业务系统维护”。这些判断在本稿中已经有了明确基础。

换句话说，页面并不是独立存在的视觉容器，而是服务能力的前台投影。首页对应的是当前计划状态、最近记录与待处理消息的聚合；计划页对应的是微计划与任务模块；记录页对应的是观察与记录模块；即时求助页对应的是 AI 会话与编排模块；消息中心则对应通知与消息模块。只要这个映射关系稳定，后续页面信息架构就可以围绕真实系统组织，而不会脱离业务闭环。

| 后续页面方向 | 本稿提供的支撑 |
|---|---|
| **首页信息架构** | 明确首页应聚合计划状态、记录入口、消息提醒与反馈回流 |
| **计划页结构** | 明确计划对象、日任务状态与周末结论都来自微计划模块 |
| **记录页结构** | 明确记录是结构化证据输入，而非孤立表单 |
| **即时求助入口** | 明确其本质是带上下文的 AI 交互，而非纯聊天 |
| **消息中心** | 明确其与推送、点击回流和状态更新之间的关系 |

## 十五、结论

到当前阶段为止，产品已经不再只是“观察内容 + 微计划内容”的组合，而可以被更清楚地描述为一套**以 iOS 为家长主入口、以后端为状态中心、以 AI 编排为生成能力、以推送为触达补充**的服务系统。这个系统的核心价值，并不在于制造更多内容，而在于把观察、支持、反馈和提醒连接成一条连续、稳定、可回流的链路。

在此基础上，后续最自然的推进方向，就是把本稿继续翻译成**页面信息架构**。届时，我们不再需要重新讨论“AI 放在哪里”“推送怎么来的”“状态由谁维护”这些底层问题，而可以直接围绕首页、计划页、记录页、消息页和即时求助页进行前台结构设计。
